# 安全模型参考

本文档提供安全模型框架的参考，用于解析设计文档中的安全要素。

---

## 4+1 视图与安全映射

4+1 视图模型（Kruchten）从不同角度描述系统架构，每个视图都有特定的安全关注点：

### 逻辑视图 → 信任边界

逻辑视图描述系统的功能分解。安全蒸馏关注：
- **模块间的信任关系**：哪些模块互相信任？信任的依据是什么？
- **接口契约**：接口的前置条件/后置条件是否包含安全校验？
- **数据流向**：敏感数据在模块间如何流转？是否经过不可信区域？

### 开发视图 → 依赖风险

开发视图描述代码的组织结构。安全蒸馏关注：
- **第三方依赖**：引入了哪些库/框架？已知漏洞情况？
- **分层架构**：各层的安全职责是否明确？是否有跨层调用绕过安全控制？
- **配置管理**：安全相关配置（认证、加密、日志）的组织方式

### 进程视图 → 并发安全

进程视图描述运行时的并发和通信。安全蒸馏关注：
- **进程间通信**：是否加密？是否有认证？
- **共享资源**：是否存在竞态条件？
- **同步机制**：锁/信号量的使用是否可能导致 DoS

### 物理视图 → 网络隔离

物理视图描述部署拓扑。安全蒸馏关注：
- **网络分区**：DMZ/内网/核心区的划分和隔离方式
- **暴露面**：哪些组件对外的网络可达？
- **数据存储**：数据库/缓存/文件存储的物理位置和访问路径

### 场景视图 → 攻击面

场景视图描述关键用例。安全蒸馏关注：
- **每个场景的攻击面**：用户交互中哪些步骤可能被利用？
- **异常流程**：错误处理路径是否安全？
- **权限提升路径**：低权限用户能否通过正常流程获取高权限？

## 价值资产分类

### 按类型

| 类型 | 示例 | 安全关注 |
|------|------|---------|
| 数据 | 用户数据、交易记录、日志 | 机密性、完整性 |
| 服务 | API、认证服务、支付网关 | 可用性、完整性 |
| 凭据 | 密码、Token、API Key | 机密性 |
| 密钥 | 加密密钥、签名密钥 | 机密性、完整性 |
| 配置 | 环境变量、策略文件 | 完整性 |

### 按分类等级

| 等级 | 定义 | 保护要求 |
|------|------|---------|
| 公开 | 可对外公开的信息 | 完整性 |
| 内部 | 仅内部使用的信息 | 机密性、完整性 |
| 机密 | 泄露会造成重大损失 | 强机密性、完整性、可审计 |
| 绝密 | 泄露会造成极严重后果 | 最高级别保护 |

## 暴露面分析框架

### 入口点类型

1. **Web API**：HTTP/HTTPS 端点，关注认证、授权、输入校验
2. **Web UI**：浏览器界面，关注 XSS、CSRF、点击劫持
3. **移动端 API**：App 通信接口，关注证书校验、请求签名
4. **消息队列**：Kafka/RabbitMQ 等，关注认证、消息完整性
5. **文件接口**：上传/下载，关注文件类型校验、路径遍历
6. **协议接口**：SSH/FTP/SMTP 等，关注认证、命令注入
7. **管理接口**：运维/监控端点，关注访问控制、默认凭据

### 信任边界

```
[互联网] ─── 信任边界1 ─── [DMZ/边缘层] ─── 信任边界2 ─── [应用层] ─── 信任边界3 ─── [数据层]
   ↑                           ↑                      ↑                       ↑
  攻击者                  反向代理/WAF            应用服务器              数据库/存储
  直接可达                部分过滤               需认证访问              内部访问
```

每个信任边界都是一个安全检查点，需要评估：
- 边界上的认证机制是否充分？
- 边界上的授权检查是否完整？
- 边界上的数据校验是否严格？
- 是否有绕过边界的路径？

## 攻击路径建模

### 路径建模原则

1. **从外到内**：每条路径从暴露面（攻击者可达）出发
2. **逐步深入**：经过信任边界，每一步标注所需条件
3. **资产导向**：路径终点是价值资产
4. **分支与决策**：标注攻击中的关键决策点和替代路径

### 路径表示示例

```
攻击路径: 认证绕过 → 数据泄露
  [入口] 公开API /api/v1/login
    │
    ├─ 步骤1: 枚举有效用户名
    │   技术: 响应差异分析
    │   条件: 登录接口对有效/无效用户名返回不同响应
    │
    ├─ 步骤2: 暴力破解弱密码
    │   技术: 字典攻击
    │   条件: 无速率限制或账户锁定机制
    │   控制: [C-01] 登录失败次数限制（有效性: 弱，可绕过）
    │
    ├─ 步骤3: 利用越权访问敏感数据
    │   技术: IDOR (不安全的直接对象引用)
    │   条件: API 接口缺乏对象级授权
    │   控制: [C-05] 对象级权限检查（有效性: 中，部分接口缺失）
    │
    └─ [目标] 用户敏感数据（机密级）
```

## 韧性控制点体系

### 按类型

| 类型 | 作用 | 示例 | 安全测试关注 |
|------|------|------|-------------|
| 预防性 | 阻止攻击发生 | WAF规则、输入校验 | 是否可绕过？ |
| 检测性 | 发现攻击行为 | IDS、日志审计 | 是否有盲区？ |
| 纠正性 | 减轻攻击影响 | 账户锁定、会话失效 | 响应是否及时？ |
| 恢复性 | 恢复正常状态 | 备份恢复、降级方案 | 恢复是否完整？ |

### 按部署层

| 层级 | 典型控制 | 测试方法 |
|------|---------|---------|
| 网络层 | 防火墙、IDS/IPS、网络隔离 | 端口扫描、协议隧道 |
| 应用层 | 认证授权、输入校验、会话管理 | 逻辑漏洞、注入、越权 |
| 数据层 | 加密、脱敏、访问控制 | 数据泄露、注入、提权 |
| 运维层 | 日志、监控、应急响应 | 日志完整性、监控覆盖 |

### 控制有效性评估

评估控制点的有效性时考虑：
- **覆盖度**：是否保护了所有相关入口点？
- **深度**：是否有纵深防御（多层控制）？
- **一致性**：类似场景的控制是否一致？
- **可绕过性**：已知或可能的绕过方式？

## STRIDE 威胁模型速查

| 威胁类型 | 安全属性 | 示例 | 测试方向 |
|----------|---------|------|---------|
| Spoofing（仿冒） | 认证 | 冒充合法用户 | 认证绕过、Token 伪造 |
| Tampering（篡改） | 完整性 | 修改传输中数据 | 参数篡改、重放攻击 |
| Repudiation（抵赖） | 不可否认性 | 否认操作行为 | 日志完整性、审计链 |
| Info Disclosure（信息泄露） | 机密性 | 暴露敏感数据 | 错误信息泄露、目录遍历 |
| DoS（拒绝服务） | 可用性 | 使服务不可用 | 资源耗尽、逻辑 DoS |
| EoP（权限提升） | 授权 | 获取更高权限 | 越权、提权链 |
